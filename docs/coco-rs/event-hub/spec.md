# Event Hub — Design Specification

> This document is the **authoritative design** for the event-hub
> feature. Implementation references this file; the round-by-round
> discussion that produced it is archived in `_discussion/` for
> historical reading only.

## 1. What we're building

A new capability for `coco-rs`: an **out-of-process event sink** that
captures the full agent event stream from one or many `coco`
processes, persists it for analysis, and serves a **richer-than-TUI
web UI** for inspection and structured-filter search.

V1 is one-way (events out of agent, into hub). The wire format is
already bidirectional (WebSocket), so Phase 3 control flow (cancel /
approve / inject) is a pure additive change.

## 2. Phase-1 scope

| In | Out |
|----|-----|
| WS-based event egress from `coco` | Auth (no bearer / mTLS / OIDC) |
| Multi-instance ingest at the hub | Phase-3 control plane |
| SQLite storage with fixed-field indexes | FTS5 / free-text search |
| `EventStore` trait — swap storage later | OTLP export |
| Desktop-first responsive web UI | Multi-hub federation |
| Mobile rendering (adaptive layout) | Cross-host TLS / cert story |
| Live updates via SSE | Performance benchmarks |
| Retention (3 days / 3 GiB defaults) | Per-instance retention overrides |
| Single-machine all-in-one mode (`--serve-hub`) | Native app shell (architecture compatible) |
| Standalone hub binary (`coco-hub-server`) | TUI hub-status chip |

### 3.1 Three crates under `coco-rs/hub/`

| Crate | Path | Role | Runs |
|-------|------|------|------|
| `coco-hub-protocol`  | `coco-rs/hub/protocol/`  | Wire types only (frames, envelope, payload, close codes) | both |
| `coco-hub-connector` | `coco-rs/hub/connector/` | Agent-side: aggregator + ring buffer + WS client | agent |
| `coco-hub-server`    | `coco-rs/hub/server/`    | Hub-side: `EventStore` trait + SQLite impl + Axum router + Web UI + SSE + retention | hub |

Why three crates: `app/cli` links protocol + connector
unconditionally; server is an optional dep behind a Cargo feature so
the default `coco` build stays small.

### 3.2 Layer placement (CLAUDE.md L0-L5 rules)

| Crate | Layer | Reason |
|-------|-------|--------|
| `coco-hub-protocol` | L1 | Depends only on `coco-types` + `serde`/`chrono`/`uuid` |
| `coco-hub-connector` | L3 | + `utils/secret-redact`, `common/otel`, `tokio-tungstenite` |
| `coco-hub-server` | L5 | + axum, rusqlite, askama, …; same layer as `coco-cli` |

## 4. Identity model

Three IDs in every event envelope:

```rust
struct EventEnvelope {
    instance_id: Uuid,    // process lifetime — constant
    session_id: String,   // rotates on /clear inside an instance
    seq: u64,             // monotonic per (instance_id, session_id)
    ts: DateTime<Utc>,    // wall clock, display only
    schema_version: u32,
    payload: EventPayload,
}
```

**`instance_id`** is generated by `Uuid::new_v4()` at `coco` startup
and persisted in `~/.coco/sessions/<pid>.json` (one new field added to
the existing `SessionRegistration` struct in
`app/session/src/concurrent_sessions.rs`). All other attributes (host,
cwd, pid, started_at, version, kind, entrypoint, name) describe the
instance but are not part of its identity — they're sent once in an
`announce` frame.

**`session_id`** rotates on `/clear` because that's verified existing
behavior (`session_runtime.rs:2706` calls `Uuid::new_v4()`). One
instance therefore contains multiple sessions in sequence:

```
instance_id = a7f3-…           (process lifetime)
  ├─ session_id = 9c19-…
  ├─ session_id = 4ee2-…       (after first /clear)
  └─ session_id = 7b18-…       (after second /clear)
```

**`seq`** is the only field the hub uses for ordering. It's assigned
by the connector aggregator (single-task) so there's no race; it
resets to 0 for each new `session_id`.

Hub URL space mirrors the hierarchy: `/i/<instance>/s/<session>`.

## 5. Wire protocol

### 5.1 Transport — WebSocket

The connector opens **one long-lived WebSocket connection per `coco`
process** to `ws[s]://<hub>/v1/connect`. All event traffic flows over
this connection. Query endpoints (Web UI, JSON API) use plain
HTTP/1.1 on the same host:port.

Why WS rather than HTTP/NDJSON: Phase-3 control plane adds frame
kinds to the same connection (no protocol upgrade). TCP backpressure
handles slow consumers. Lower per-batch overhead.

### 5.2 Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| **WS** | `ws[s]://<hub>/v1/connect` | One WS per connector; all event traffic |
| `GET` | `/v1/instances` | List known instances |
| `GET` | `/v1/instances/{instance_id}` | One instance + attributes |
| `GET` | `/v1/instances/{instance_id}/sessions` | List sessions |
| `GET` | `/v1/instances/{instance_id}/sessions/{session_id}/events` | Paged events |
| `GET` | `/v1/search` | Structured-filter search (no `q` in V1) |
| `GET` | `/healthz` | Liveness |
| `GET` | `/v1/protocol` | Supported subprotocol versions |
| Web pages + HTMX partials | `/`, `/i/*`, `/p/*`, `/sse/*` | See §7 |

### 5.3 Subprotocol negotiation

Standard WebSocket subprotocol header:

```
Client → Server:  Sec-WebSocket-Protocol: coco-event-hub.v1
Server → Client:  Sec-WebSocket-Protocol: coco-event-hub.v1
```

No overlap → handshake fails with HTTP 400 from the server. Per-
variant `schema_version` (§5.5) lets us add fields without bumping
the subprotocol.

### 5.4 Frame kinds

Every WS text frame on `/v1/connect` carries one JSON object with a
`kind` discriminator:

| `kind` | Direction | Purpose |
|--------|-----------|---------|
| `announce` | client → hub | First frame; carries instance_id + attributes |
| `announce_ack` | hub → client | Response: `first_seen`, `hub_version`, `resume_from` |
| `batch` | client → hub | A batch of envelopes: `{ events: [...] }` |
| `batch_ack` | hub → client | Optional: `up_to_seq` durable in storage (V1 doesn't block on it) |
| `error` | hub → client | Pre-close diagnostic: `code`, `detail` |
| *(future)* `control_*` | hub → client | Phase-3 control plane |

Unknown `kind` from either side: logged + ignored, not close-worthy.

### 5.5 Event envelope

```json
{
  "instance_id": "a7f3...",
  "session_id": "9c19...",
  "seq": 12345,
  "ts": "2026-05-16T12:34:56.789Z",
  "schema_version": 1,
  "payload": { "kind": "...", ... }
}
```

### 5.6 Payload variants

Tagged union (`#[serde(tag = "kind", rename_all = "snake_case")]`):

| `kind` | Source | Description |
|--------|--------|-------------|
| `protocol` | `CoreEvent::Protocol(ServerNotification)` pass-through | All lifecycle events (turn / item / subagent / MCP / compaction / hooks / …) — keeps `ServerNotification`'s internal `type` tag |
| `tool_use_queued` / `tool_use_started` / `tool_use_completed` | `CoreEvent::Stream` tool variants | Tool lifecycle; full input/output after `utils/secret-redact` |
| `mcp_tool_call_begin` / `mcp_tool_call_end` | `CoreEvent::Stream` MCP variants | Same shape, tagged for MCP |
| `text_block_completed` / `thinking_block_completed` | Aggregated from per-token deltas | One row per content block: `{ turn_id, agent_id, block_index, full_text, char_count, started_at, completed_at, was_truncated? }` |
| `events_dropped` | Connector-synthesized | `{ count, since_seq, until_seq, reason: "ring_buffer_overflow" }` — emitted when the ring drops events |

`CoreEvent::Tui(...)` is **never sent** — it's terminal-only and
worthless in a web UI.

Unknown `kind` at the hub: stored as `parse_status = "unknown_kind"`
with the original JSON; UI renders as "(unknown event: kind=X)".

### 5.7 Aggregation state machine

The connector accumulates per-token text/thinking deltas per
`(turn_id, item_id)` and flushes one `text_block_completed` /
`thinking_block_completed` envelope at the natural boundary. This is
the **single non-trivial transform** in the connector.

State per `(turn_id)`:

```rust
struct TurnAggregator {
    turn_id: String,
    next_block_index: u32,
    open: Option<OpenBlock>,
}
struct OpenBlock {
    kind: TextOrThinking,
    item_id: String,
    block_index: u32,
    started_at: DateTime<Utc>,
    buffer: String,
    agent_id: Option<AgentId>,
}
```

Transitions:

| Input event | Action |
|-------------|--------|
| `Protocol(TurnStarted)` | Allocate aggregator. Forward as `protocol`. |
| `Protocol(ItemStarted { kind: text\|thinking })` | If open ≠ None, flush defensively. Open new block. Forward `ItemStarted` as `protocol`. |
| `Stream(TextDelta)` | Append to `open.buffer` (silent drop if open.kind ≠ text). **Do not forward delta to wire.** |
| `Stream(ThinkingDelta)` | Append (silent drop if open.kind ≠ thinking). Do not forward. |
| `Protocol(ItemCompleted)` | Emit `text_block_completed` / `thinking_block_completed`. Close block. Forward `ItemCompleted` as `protocol`. |
| `Stream(ToolUse*)` / `Stream(McpToolCall*)` | Forward 1:1. |
| `Protocol(TurnCompleted/Failed/Interrupted)` | If `open` still set, flush with `was_truncated: true`. Drop aggregator. Forward terminator. |
| `Tui(_)` | Silently drop. No seq consumed. |
| Any other `Protocol(_)` | Forward as `protocol`. |

Guarantees: one block per content boundary, interleaved with tool
events via shared `seq`; per-token deltas never reach the wire;
mid-turn crash flushes whatever buffer exists with truncation marker.

Volume reduction vs raw stream: ~18× (typical 30-turn / 5-tool /
500-token session).

### 5.8 Batching & reconnect

The connector accumulates envelopes in a bounded ring buffer (size
`event_hub_ring_buffer_size`, default 10 000) and emits a `batch`
frame when any of:

- ≥ `event_hub_batch_max_events` (default 1000) pending
- Pending serialized bytes ≥ `event_hub_batch_max_bytes` (default 1 MiB)
- ≥ `event_hub_batch_max_interval_ms` (default 500 ms) since last send
- Ring buffer ≥ 80 % full
- WS just (re)connected and backlog exists
- Shutdown (final synchronous flush, 2 s deadline)

If a single envelope exceeds the batch byte limit, it goes in a
single-event frame. Hub's hard frame ceiling is
`hub_max_frame_bytes` (default 10 MiB); above that the hub closes
with code `4013`.

The connector has three internal tasks: aggregator (consumes
`CoreEvent`), sender (owns WS write half), reader (owns WS read half;
consumes ack/error frames).

**Reconnect:** any broken connection → mark reconnecting, push
in-flight batch back to ring buffer front, exponential backoff
(100 ms → 200 ms → … → 30 s, ±20 % jitter), reopen WS, resend
`announce`, drain from oldest seq. Hub deduplicates via
`(instance_id, session_id, seq)` primary key. If ring buffer
overflows during reconnect, drop oldest + enqueue `events_dropped`
marker.

Non-retriable close codes (`4000`, `4001`, `4002`): stop retrying on
the same config; log and wait for reconfiguration.

### 5.9 Close codes

| Code | Meaning | Connector action |
|------|---------|------------------|
| `1000` Normal Closure | Hub shutdown | Reconnect with backoff |
| `1001` Going Away | Hub restart | Reconnect with backoff |
| `1011` Internal Error | Hub bug | Reconnect with backoff |
| `4000` Bad Request | Malformed announce / frame | Do not retry |
| `4001` Protocol Mismatch | No subprotocol overlap | Do not retry until reconfigured |
| `4002` Auth Failed | **Reserved** — V1 hub never emits | Do not retry until reconfigured |
| `4013` Frame Too Large | Frame > `hub_max_frame_bytes` | Do not retry (config error) |
| `4029` Rate Limited | Hub backpressure | Reconnect with longer backoff |

Before any `4xxx` close, the hub sends one `error` frame with
`{ code, detail }` for diagnostics.

## 6. Storage

### 6.1 Backend: SQLite

Single file at `<hub_data_dir>/events.sqlite`, WAL mode,
`synchronous=NORMAL`. One writer task owns the connection; reads
through a `deadpool-sqlite` pool (default 8).

V1 has **fixed-field indexes only** — no FTS5. Free-text search is
deferred to V2; the `EventStore::search` trait already accepts a `q:
Option<String>` field but the V1 SQLite impl returns
`NotSupported` if it's set, and the HTTP `/v1/search` endpoint
returns 400 with `{ "error": "free_text_not_supported" }`.

### 6.2 `EventStore` trait — storage interface abstraction

All storage access goes through an `async` trait (the file lives in
`coco-rs/hub/server/src/store/mod.rs`; not its own crate). Swapping
SQLite for Postgres / DuckDB later is one feature flip plus an
`Arc<dyn EventStore>` constructor change in `coco-hub-server::main`.

```rust
#[async_trait::async_trait]
pub trait EventStore: Send + Sync + 'static {
    async fn upsert_instance(&self, announce: &AnnounceFrame) -> Result<UpsertInstanceOutcome>;
    async fn mark_instance_gone(&self, instance_id: &str, reason: GoneReason) -> Result<()>;
    async fn ingest_batch(&self, instance_id: &str, batch: BatchFrame) -> Result<IngestStats>;

    async fn list_instances(&self, params: ListInstancesParams) -> Result<Page<InstanceRow>>;
    async fn get_instance(&self, instance_id: &str) -> Result<Option<InstanceRow>>;
    async fn list_sessions(&self, instance_id: &str, params: ListSessionsParams) -> Result<Page<SessionRow>>;
    async fn get_session(&self, instance_id: &str, session_id: &str) -> Result<Option<SessionRow>>;
    async fn list_events(&self, query: EventQuery) -> Result<Page<EventRow>>;
    async fn get_event(&self, instance_id: &str, session_id: &str, seq: i64) -> Result<Option<EventRow>>;
    async fn search(&self, query: SearchQuery) -> Result<Page<SearchHit>>;
    async fn list_agent_edges(&self, instance_id: &str, session_id: &str) -> Result<Vec<AgentEdge>>;
    async fn run_retention_sweep(&self, policy: &RetentionPolicy) -> Result<SweepStats>;
    async fn health(&self) -> Result<HealthSnapshot>;
}
```

Boundary rules:

- Server holds `Arc<dyn EventStore>`; no raw `rusqlite::Connection`
  outside the SQLite impl.
- Row types (`InstanceRow`, `SessionRow`, `EventRow`, `AgentEdge`,
  `SearchHit`) are backend-agnostic structs.
- Cursors are opaque `String` (base64 by convention); impls encode
  their own format.
- Live SSE fanout is **not** in the trait — that's a hub-internal
  `broadcast::Sender<EventEnvelope>` per session topic, kept
  separate from durable storage.
- `MockEventStore` is feature-gated (`test-util`).

### 6.3 Schema (SQLite)

```sql
-- 6.3.a  Instances
CREATE TABLE instances (
    instance_id    TEXT PRIMARY KEY,
    host           TEXT NOT NULL,
    cwd            TEXT NOT NULL,
    pid            INTEGER NOT NULL,
    started_at     INTEGER NOT NULL,       -- unix ms
    version        TEXT NOT NULL,
    kind           TEXT NOT NULL,          -- interactive | bg | daemon | daemon_worker
    entrypoint     TEXT,
    name           TEXT,
    first_seen_at  INTEGER NOT NULL,
    last_seen_at   INTEGER NOT NULL,
    status         TEXT NOT NULL DEFAULT 'unknown'
);
CREATE INDEX idx_instances_host_cwd  ON instances(host, cwd);
CREATE INDEX idx_instances_last_seen ON instances(last_seen_at DESC);
CREATE INDEX idx_instances_status    ON instances(status);

-- 6.3.b  Sessions (derived from event stream, not announced separately)
CREATE TABLE sessions (
    instance_id   TEXT NOT NULL,
    session_id    TEXT NOT NULL,
    started_at    INTEGER,
    ended_at      INTEGER,
    model         TEXT,
    total_input_tokens   INTEGER DEFAULT 0,
    total_output_tokens  INTEGER DEFAULT 0,
    total_cost_usd       REAL    DEFAULT 0,
    last_seq      INTEGER DEFAULT 0,
    last_event_ts INTEGER,
    discovered_via TEXT NOT NULL DEFAULT 'session_started',
    PRIMARY KEY (instance_id, session_id),
    FOREIGN KEY (instance_id) REFERENCES instances(instance_id) ON DELETE CASCADE
);
CREATE INDEX idx_sessions_started    ON sessions(started_at DESC);
CREATE INDEX idx_sessions_last_event ON sessions(last_event_ts DESC);

-- 6.3.c  Events
CREATE TABLE events (
    instance_id    TEXT NOT NULL,
    session_id     TEXT NOT NULL,
    seq            INTEGER NOT NULL,
    ts             INTEGER NOT NULL,       -- envelope.ts (agent clock)
    received_at    INTEGER NOT NULL,       -- hub clock
    schema_version INTEGER NOT NULL,
    kind           TEXT NOT NULL,

    -- denormalized for indexed search; NULL when N/A:
    turn_id        TEXT,
    agent_id       TEXT,
    item_id        TEXT,
    tool_name      TEXT,
    call_id        TEXT,
    is_error       INTEGER,
    inner_kind     TEXT,

    payload        TEXT NOT NULL,          -- raw envelope JSON
    payload_size   INTEGER NOT NULL,
    parse_status   TEXT NOT NULL DEFAULT 'ok',

    tool_output_truncated  INTEGER NOT NULL DEFAULT 0,
    tool_output_full_size  INTEGER,

    preview        TEXT,                   -- UI snippet, NOT indexed

    PRIMARY KEY (instance_id, session_id, seq),
    FOREIGN KEY (instance_id, session_id)
        REFERENCES sessions(instance_id, session_id) ON DELETE CASCADE
);
CREATE INDEX idx_events_session_ts  ON events(instance_id, session_id, ts);
CREATE INDEX idx_events_kind        ON events(kind);
CREATE INDEX idx_events_tool_name   ON events(tool_name) WHERE tool_name IS NOT NULL;
CREATE INDEX idx_events_errors      ON events(instance_id, session_id, seq) WHERE is_error = 1;
CREATE INDEX idx_events_received_at ON events(received_at);
CREATE INDEX idx_events_agent       ON events(instance_id, session_id, agent_id) WHERE agent_id IS NOT NULL;

-- 6.3.d  Sub-agent topology (re-derived at ingest from SubagentSpawned events)
CREATE TABLE agent_edges (
    instance_id      TEXT NOT NULL,
    session_id       TEXT NOT NULL,
    parent_agent_id  TEXT NOT NULL,
    child_agent_id   TEXT NOT NULL,
    agent_type       TEXT,
    spawned_at       INTEGER NOT NULL,
    completed_at     INTEGER,
    status           TEXT NOT NULL DEFAULT 'running',
    PRIMARY KEY (instance_id, session_id, child_agent_id),
    FOREIGN KEY (instance_id, session_id)
        REFERENCES sessions(instance_id, session_id) ON DELETE CASCADE
);
CREATE INDEX idx_agent_edges_parent ON agent_edges(instance_id, session_id, parent_agent_id);
CREATE INDEX idx_agent_edges_status ON agent_edges(status);
```

`events_fts` (FTS5) is **not created in V1**. Reserved for V2.

### 6.4 Ingest pipeline

Per WS `batch` frame, one transaction:

1. `denormalize(envelope)` → fills the indexed columns above.
2. Insert into `events`; duplicate PK = silent no-op (retry artifact).
3. If payload is text-bearing, compute `preview` (first 200 chars).
4. Update `sessions` rollup (`last_seq`, `last_event_ts`, optional
   token/cost rollup).
5. If `protocol` event with `inner_kind == "subagent_spawned"`,
   upsert into `agent_edges`. Subagent completion / backgrounding /
   final progress → update `agent_edges.status`.
6. Update `instances.last_seen_at`.
7. Commit.

Backpressure: if writer queue ≥ 1 000 batches deep, WS reader pauses
reading; resumes when queue < 500. Propagates as TCP-level
backpressure to the agent.

After the storage write, the ingest pipeline also publishes each
envelope to the per-session `broadcast::Sender<EventEnvelope>` for
SSE fanout (§7.3).

### 6.5 Tool output handling

- Connector sends full tool outputs inline after `utils/secret-redact`.
- Hub truncates anything above `hub_max_tool_output_inline` (default
  256 KiB) and records `tool_output_truncated = 1` +
  `tool_output_full_size`. Truncated payload contains an explicit
  marker (e.g. `"output_text": "…[truncated 250 KiB]"`).
- WS frame hard ceiling is `hub_max_frame_bytes` (default 10 MiB).
  Above that → close code 4013.
- No blob storage in V1. Schema has fields ready for V2.

### 6.6 Retention

Defaults: **3 days / 3 GiB**.

| Knob | Default |
|------|---------|
| `hub_retention_days` | 3 |
| `hub_retention_max_bytes` | 3 221 225 472 (3 GiB) |
| `hub_retention_sweep_interval_secs` | 900 (15 min) |
| `hub_vacuum_threshold_bytes` | 268 435 456 (256 MiB) |

Background tokio task every `sweep_interval`:

1. Delete events where `received_at < now - retention_days * 86_400_000`.
2. If DB size still over cap, delete oldest complete sessions until
   under cap. ("Complete" = `ended_at IS NOT NULL` OR last event >
   24 h ago.)
3. `PRAGMA incremental_vacuum;` after each sweep.
4. Full `VACUUM` opportunistically when cumulative freed pages
   exceed `vacuum_threshold` and no writes happened in the last 60 s.

## 7. Web UI

### 7.1 Stack

Server-rendered HTML, no SPA. The HTTP surface is the contract; HTML
is one consumer of it.

| Layer | Tech | Why |
|-------|------|-----|
| HTTP | **Axum** | Same stack hosts WS, JSON API, HTML pages |
| Templates | **Askama** | Compile-time-checked HTML |
| Interactivity | **HTMX** | `hx-get`/`hx-swap` for partial updates; `hx-ext="sse"` for live; works in any browser including mobile WebView |
| Styling | **Tailwind CSS v4** | Responsive utilities; CSS-first config (no JS config file); standalone CLI |
| Components | **Flowbite** | Pre-styled Tailwind components (tables, modals, drawers) |
| Live updates | **SSE** | One-way push; HTMX has native support; cheaper than per-tab WS |
| Syntax highlighting | **Prism.js** + 11 language bundles | JSON, Bash, Python, JS, TS, Rust, Go, YAML, Markdown, SQL, Diff |

Forward-compat with a future native app shell: WebView wraps the
same HTML pages, or a native UI calls the JSON API at `/v1/*`.

### 7.3 SSE plumbing

```
WS reader → ingest_batch() →┬→ EventStore::ingest_batch (SQLite)
                            └→ broadcast::Sender<EventEnvelope> per session topic
                                          └→ SSE subscriber → Askama partial → SSE data: line
```

Each browser tab opens one SSE connection per session it's viewing.
Browser close → subscriber drops → no server-side reconnect
bookkeeping. HTMX `hx-ext="sse"` + `sse-connect="/sse/session/.../..."`
+ `sse-swap="event"` swaps the fragment into the event list.

### 7.4 Search API surface

`GET /v1/search` accepts structured filters only in V1. If `q` is
set, returns HTTP 400 with body
`{ "error": "free_text_not_supported", "detail": "..." }`.

Parameters:

| Param | Type | Meaning |
|-------|------|---------|
| `instance` | string | Filter to one instance |
| `session` | string | Filter to one session (requires `instance`) |
| `agent` | string | Filter by `agent_id` |
| `kind` | string | Outer kind (`protocol`, `tool_use_completed`, …) |
| `inner_kind` | string | Inner Protocol kind |
| `tool` | string | Tool name |
| `error` | `0`/`1` | Filter by `is_error` |
| `from` / `to` | RFC 3339 | Time range |
| `limit` | int | 1..500, default 50 |
| `cursor` | opaque | base64 `{seq, ts}` for stable ordering |

When V2 adds FTS, `q` simply starts being honored; the response shape
is unchanged.

### 7.5 Responsive design — desktop-first, adaptive

Design baseline is a desktop browser (≥ 1280 px). Mobile is a
fallback rendering of the same content, not a separately designed
surface.

| Breakpoint | Layout |
|-----------|--------|
| ≥ 1024 px (`lg:`) | Primary: full table layouts, sidebar visible |
| 640–1024 px (`sm:` / `md:`) | Condensed tables, sidebar may collapse |
| < 640 px | Single column, sidebar = Flowbite off-canvas drawer |

No mobile-only features (swipe, pull-to-refresh). No client-side
virtualization — server-side cursor pagination via HTMX.

### 7.6 Static assets

Committed in `coco-rs/hub/server/web/static/`. Versions pinned:

| Asset | Version |
|-------|---------|
| Flowbite (CSS + JS) | v2.x latest |
| HTMX | v1.9.x latest |
| HTMX SSE extension | matching HTMX |
| Prism.js core | v1.29.x latest |
| Prism languages | v1.29.x — 11 bundles: json, bash, python, javascript, typescript, rust, go, yaml, markdown, sql, diff |
| Prism theme | `prism-tomorrow` v1.29.x |

Tailwind CSS is **built** by `build.rs` invoking the `tailwindcss`
v4 CLI on `web/style.css`; output `include_bytes!`'d into the binary.

Updates via manual bump PR (download new minified file, replace,
verify, PR).

## 8. Configuration

### 8.1 Connector side (`event_hub_*` keys)

Settings keys, env vars (`COCO_EVENT_HUB_*`), and CLI flags
(`--event-hub-url`) all flat at root. Presence of `event_hub_url` =
connector enabled.

| Key | Default | Meaning |
|-----|---------|---------|
| `event_hub_url` | `null` | `ws[s]://host:port/v1/connect`; `null` ⇒ disabled |
| `event_hub_bearer_token` | `null` | Reserved (V1 ignored) |
| `event_hub_ring_buffer_size` | 10 000 | Max envelopes in memory |
| `event_hub_batch_max_events` | 1 000 | Max events per `batch` frame |
| `event_hub_batch_max_bytes` | 1 048 576 (1 MiB) | Max bytes per frame |
| `event_hub_batch_max_interval_ms` | 500 | Max flush delay |

### 8.2 Hub side (`hub_*` keys)

| Key | Default | Meaning |
|-----|---------|---------|
| `hub_data_dir` | `~/.coco/hub/` (embedded) or `$PWD/data/` (standalone) | SQLite + asset dir |
| `hub_bind_addr` | `127.0.0.1` | Bind address |
| `hub_port` | 8731 | Port |
| `hub_max_frame_bytes` | 10 485 760 (10 MiB) | Reject frames above |
| `hub_max_tool_output_inline` | 262 144 (256 KiB) | Truncate beyond |
| `hub_retention_days` | 3 | |
| `hub_retention_max_bytes` | 3 GiB | |
| `hub_retention_sweep_interval_secs` | 900 (15 min) | |
| `hub_vacuum_threshold_bytes` | 256 MiB | |
| `hub_log_level` | `info` | tracing filter |

No auth knob in V1.

### 8.3 CLI flags (`coco`)

| Flag | Effect |
|------|--------|
| `--event-hub-url <URL>` | Overrides `event_hub_url`; must be `ws://` or `wss://` |
| `--serve-hub` | All-in-one: spawn hub in same process; auto-sets `event_hub_url = ws://127.0.0.1:<hub_port>/v1/connect`. Requires `serve-hub` Cargo feature compiled in. |
| `--hub-port <N>` | Port for `--serve-hub` (default 8731) |

When the `serve-hub` feature is off and `--serve-hub` is invoked:

> *"This `coco` build was not compiled with the `serve-hub` feature.
> Rebuild with `just coco-with-hub` or `cargo build -p coco-cli
> --features serve-hub`. Alternatively, run a separate
> `coco-hub-server` process and point `event_hub_url` at it."*

### 8.4 Standalone hub binary CLI

```
coco-hub-server serve
  --bind 127.0.0.1
  --port 8731
  --data-dir ./data
  [--dev-assets ./coco-rs/hub/server/web]  # dev only; reads CSS/JS from disk
```

## 9. Build

### 9.1 Cargo features

```toml
# coco-rs/app/cli/Cargo.toml
[features]
default   = []                              # default = no hub server
serve-hub = ["dep:coco-hub-server"]

[dependencies]
coco-hub-protocol  = { path = "../../hub/protocol" }
coco-hub-connector = { path = "../../hub/connector" }
coco-hub-server    = { path = "../../hub/server", optional = true }
```

Default `cargo build` does **not** include the hub server (no axum /
sqlite / askama / Tailwind asset build). Opt in via `--features
serve-hub`.

### 9.2 `justfile` recipes

Appended to `coco-rs/justfile`:

```just
# Default coco binary — NO embedded hub server.
# This is the only build verified by `just pre-commit`.
coco:
    cargo build -p coco-cli

# coco with the embedded hub server (all-in-one mode).
# Requires `tailwindcss` v4 CLI in $PATH (or COCO_TAILWIND_CLI env).
# Quality NOT gated by pre-commit — run manually after touching hub crates.
coco-with-hub:
    cargo build -p coco-cli --features serve-hub

# Standalone hub-server binary.
hub-server:
    cargo build -p coco-hub-server
```

Existing recipes (`fmt`, `quick-check`, `pre-commit`, `test`,
`test-crate`, `fix`, `clippy`, `check`, `help`) unchanged.
`just pre-commit` only verifies the default build.

### 9.3 Workspace structure

`coco-rs/Cargo.toml`:

```toml
[workspace]
resolver = "2"
members = [
    # ... existing crates unchanged ...
    "hub/protocol",
    "hub/connector",
    "hub/server",
]
default-members = ["app/cli"]
```

| Command | Builds |
|---------|--------|
| `cargo build` (root) | only `coco-cli` |
| `cargo build --workspace` | all members |
| `cargo build -p coco-hub-server` | standalone hub |

### 9.4 Tailwind v4 CLI policy

Contributors install `tailwindcss` v4 themselves. No vendoring, no
network download.

| Platform | Source |
|----------|--------|
| macOS | `brew install tailwindcss` |
| Linux | Download from https://github.com/tailwindlabs/tailwindcss/releases |
| Windows | `winget install Tailwindcss.Tailwindcss` |

`build.rs` detection: `COCO_TAILWIND_CLI` env → `which::which("tailwindcss")`.
If missing, fails with a clear "install tailwindcss v4 from …, or set
COCO_TAILWIND_CLI" error. Only `--features serve-hub` builds touch
this.

### 9.6 Dev-loop

Two terminals, both from `coco-rs/`:

```
# T1 — Rust rebuild on source change
cargo watch -x 'run -p coco-hub-server -- serve --dev-assets coco-rs/hub/server/web --port 8731'

# T2 — Tailwind rebuild on template/style.css change
tailwindcss -i coco-rs/hub/server/web/style.css \
            -o coco-rs/hub/server/web/static/dev-style.css \
            --watch
```

`--dev-assets <dir>` flag on the hub server reads CSS/JS/templates
from disk instead of the embedded copy; templates still require Rust
rebuild (Askama is compile-time), but CSS hot-reloads. Detailed in
`coco-rs/hub/server/CLAUDE.md`.

## 10. Implementation prerequisites (mechanical)

These don't warrant design discussion; listed so they're not
forgotten.

- **`EnvKey` registrations** in `coco-rs/common/config/src/`: 6
  `COCO_EVENT_HUB_*` keys (connector) + 10 `COCO_HUB_*` keys (server).
  `COCO_TAILWIND_CLI` is read by `build.rs` only, not in `EnvKey`.
- **`StatusCode` allocations** in `coco-rs/common/error/`: category
  `EventHub = 14`. Sub-codes:
  | Code | Variant |
  |------|---------|
  | 14_001 | `ConnectorSendFailed` |
  | 14_002 | `ConnectorBufferOverflow` |
  | 14_003 | `HubProtocolMismatch` |
  | 14_004 | `HubFrameTooLarge` |
  | 14_005 | `HubInvalidAnnounce` |
  | 14_006 | `StoreError` |
  | 14_007 | `StoreNotSupported` (V1 free-text path) |
  | 14_008 | `TailwindCliMissing` (build-time) |
- **Per-crate `CLAUDE.md`** for each new crate (project convention).
- **Root `coco-rs/CLAUDE.md`** gains a Hub group in the crate table
  and a reference to this `docs/coco-rs/event-hub/spec.md` under
  Specialized Documentation.
- **`SessionRegistration`** in
  `coco-rs/app/session/src/concurrent_sessions.rs` gains
  `instance_id: Uuid`.
- **`QueryEngine`** (or wherever the `CoreEvent` `Sender` is owned)
  clones one additional sender for `coco-hub-connector` when
  `event_hub_url` is set.

## 11. Phase-1 Definition of Done

**Hard gate (CI):**

- [ ] `just coco` succeeds on macOS (Apple Silicon + Intel) and Linux x86_64.
- [ ] `just pre-commit` passes.
- [ ] `coco --serve-hub` (without `serve-hub` feature) prints the
      documented diagnostic and exits non-zero.

**Manual verification (operator runs through):**

- [ ] `just coco-with-hub` succeeds; binary size delta within
      ~10–20 MiB of the default.
- [ ] `cargo test -p coco-hub-protocol -p coco-hub-connector -p coco-hub-server` succeeds.
- [ ] Single-machine all-in-one: `coco --serve-hub` runs ≥ 5 min
      idle; web UI renders in < 1 s; sending a TUI prompt makes the
      session appear in the web UI; events stream live via SSE; tool
      calls show name + input/output preview; mobile viewport
      collapses to single column; Prism highlights JSON/Bash/Diff.
- [ ] Multi-instance: standalone `coco-hub-server` + two `coco`
      processes in different cwds → web UI shows both with distinct
      attributes; structured-filter search works across instances.
- [ ] Resilience: kill hub mid-session → agent keeps running →
      restart hub → reconnects → no events lost or `EventsDropped`
      marker visible in UI; `/clear` rotates `session_id` and a new
      session row appears under the same `instance_id`.
- [ ] Retention sweep (lowered interval, backdated rows) deletes
      expected rows and runs `incremental_vacuum`. Defaults don't
      delete anything during the smoke test.
- [ ] Search API: `tool=Shell` returns only Shell events; `error=1`
      only failures; `q=foo` returns 400 with documented body.

(lowest risk, highest value):
